## 一、整体架构

飞书生态分三层：

```
┌─────────────────────────────────────┐
│  用户层：飞书 App（移动/桌面）         │
├─────────────────────────────────────┤
│  开放平台：2500+ API + 事件订阅       │
│  - 自建应用（企业内部）               │
│  - ISV 应用（商店上架）               │
├─────────────────────────────────────┤
│  底层：认证（tenant token / user token）│
└─────────────────────────────────────┘
```

**关键概念：**
- `tenant_access_token` = 机器人身份（我用的）
- `user_access_token` = 代表用户操作（需 OAuth）
- 两者权限范围完全不同，不能混用

## 三、IM 消息机制（Channel Binding）

### 消息类型
- `text`：纯文本，≤150KB
- `post`：富文本，≤30KB
- `image/file/audio/media`：`file_key`引用
- `interactive`：卡片消息，≤30KB，支持流式更新

### sessionKey 建立规则

```
私聊（p2p）：       peerId = senderOpenId
群聊（group）：     peerId = chatId（oc_xxx）
  - 默认：整个群一个 session
  - 可选：每人独立（group_sender）
  - 可选：话题独立（group_topic）
```

**重要发现：** 群聊默认所有成员共享同一个 AI 对话上下文。如果太擎要给群里不同人不同的 AI 助手体验，需要改成 `group_sender` 模式。

### 核心坑点
1. 消息可能重复推送——必须用 `message_id` 去重
2. 编辑（PUT）≠ 更新卡片（PATCH）——接口不能混用
3. 发消息给用户时，用户必须在**应用可用范围**内
4. 5 QPS 限制（向同一用户/群发消息）

## 四、商业机会（太擎 × 飞书）

### 🥇 机会 1：财税知识库机器人（**最快落地**）
- 用 `feishu_wiki` + `feishu_doc` 建政策文件知识库
- 用户飞书单聊提问 → AI 读文档回答
- **现在就能做**

### 🥈 机会 2：财务报告自动生成（**次快落地**）
- 财务数据录入 `feishu_bitable`
- AI 定期读取（`list_records`）→ 分析 → 写报告到 `feishu_doc`
- 推送链接到群

### 🥉 机会 3：报销审批 × AI 审核
- 员工提交报销 → 飞书审批流触发 → AI 审核票据合规性 → 返回意见
- **难点：需要飞书审批 API（企业管理员特权）**，不是普通开发者能拿到的

### 机会 4：文档智能标签 + 检索
- 批量处理 Wiki 节点内容 → AI 自动打标签/摘要
- 现有工具可实现，但批量循环需自己写

## 五、关键结论

**关于「掌控飞书」：**

以 OpenClaw 现有插件 + 108 个已授权 scope，我对飞书的掌控程度：

- ✅ **读**：文档、知识库、多维表格、文件夹、群信息
- ✅ **写**：文档创建/更新、知识库节点、多维表格记录
- ✅ **发**：飞书消息（私聊/群聊）
- ❌ **管**：删除（节点/记录/字段）、批量操作、权限细控
- ❌ **触**：审批流触发、Bitable 自动化事件（需框架扩展）

**一句话结论：飞书的「内容层」已经掌控较为完整，「管理层」和「自动化层」是缺口。**
