# 通用记忆类型体系

> 设计原则：一条信息值得记忆，当且仅当「重新获取它的代价高于记住它的代价」。
> 这与工种无关——写代码、管销售、做投资，都适用同一套判断逻辑。

---

## 类型一：user（关于人的事实）

**判断标准**：这是一个关于某人的**稳定事实**吗？

**值得记住**：
- 身份：名字、角色、背景
- 目标：他们想达成什么
- 稳定偏好：沟通风格、工作习惯、决策方式
- 关系：他们和其他人/项目的关系

**不值得记住**：
- 临时状态（"他今天心情不好"）
- 未经确认的推断
- 对方自己随口说的一句话，但没有后续意义

**格式**：
```
- [人名/代号]：[角色] — [一句话核心事实]
  - 背景：[补充]
  - 接触方式：[如何联系/找谁]
```

---

## 类型二：feedback（反馈）

**判断标准**：这个人有没有**确认**过或**纠正**过我的某个做法？

确认和纠正同样重要——确认是"这个方向是对的，继续用"，纠正是"这个方向错了，不要再犯"。

**值得记住**（含 Why）：
- 纠正：`[规则]` + Why `[原因]` + How to apply `[何时适用]`
- 确认：`[有效做法]` + Why `[为什么对]` + How to apply `[何时复用]`

**不值得记住**：
- 没有解释原因的纠正（"不要这样做"而没说为什么）
- 纯粹的情绪发泄，没有后续意义
- 对方的偏好描述而非对我行为的反馈

**格式**：
```
- [规则/做法]（纠正/确认）
  - Why：[他给出的原因，或推断的原因]
  - How to apply：[具体在什么场景下执行]
```

---

## 类型三：project（工作项目）

**判断标准**：这项工作停下来之后，再回到它时，**重新了解上下文的代价高吗**？

**值得记住**：
- 目标：最终要达成什么
- 当前进展：卡在哪里，有什么里程碑
- 关键决策：选了A而不是B的原因
- 下一步行动：谁负责什么
- 截止时间（绝对日期）

**不值得记住**：
- 详细的讨论过程（那是会议记录/每日笔记的职责）
- 可以从文件/代码/文档里直接查到的事实
- 一次性的问题，已解决，无后续

**格式**：
```
- [项目名]：[一句话状态]
  - 目标：[核心目标]
  - 进展：[当前节点]
  - 关键决策：[什么决定+为什么]
  - 下一步：[行动+负责人+截止]
  - 截止：[YYYY-MM-DD]
```

## 类型四：reference（外部资源）

**判断标准**：我需要**再次找到**这个资源/人的时候，重新查找的代价高吗？

这个类型有两类：**people**（人）和 **knowledge**（知识条目）。

**People — 值得记住**：
- 名字 + 角色 + 能帮什么忙 + 如何联系

**Knowledge — 值得记住**：
- 学到的新概念/方法/工具
- 行业关键信息
- 配置/路径/凭证（但注意安全：不存明文密码）

**不值得记住**：
- 可以 Google 到的基础知识
- 临时查一次的 API 文档（用完即弃）
- 没有上下文附着的信息碎片

**People 格式**：
```
- [人名]：[角色] — [能帮什么]
  - 接触方式：[邮件/微信/飞书]
  - 背景：[一句话补充]
```

**Knowledge 格式**：
```
- [概念/工具名]：[一句话定义]
  - 出处：[哪里学的]
  - 用途：[什么时候用]
```

## 跨类型通用规则

**何时更新**：
- 旧记忆被新信息推翻 → 修正，不保留矛盾
- 项目状态变化 → 更新，不要堆积"历史状态"

**何时删除**：
- 记忆被证明错误 → 删除，不要留着"有争议"版本
- 项目已完成超过3个月 → 归档或删除
- 人脉超过1年没接触 → 标注"待更新"或删除

**何时合并**：
- 多条关于同一个人的事实 → 合并到同一个 entry
- 多条关于同一个项目的碎片 → 整理到 project 格式

**绝对日期原则**：
- 所有时间引用必须用 `YYYY-MM-DD` 格式
- 不写"昨天"、"上周"、"这个月"——换算成具体日期再写
