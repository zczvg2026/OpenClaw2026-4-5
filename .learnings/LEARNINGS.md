## 2026-03-25

### 错误：未遵循渐进式迭代

**问题描述**：
用户要求审阅政府报告并给出修改建议。我给出了建议后，用户说"按这个改吧"，我直接输出了完整报告并创建飞书文档，没有分步确认。

**正确流程应该是**：
1. 先给修改建议框架 → 用户确认
2. 用户确认后，给出完整内容 → 用户确认
3. 用户确认后，再创建飞书文档

**为什么会犯**：
- 惯性思维：想快速完成交付
- 没有主动问用户想要哪种确认方式
- 忘记"渐进式迭代"的核心理念：小步快跑，持续确认

**教训**：
- 任何需要用户确认的产出，先给框架/方向
- 等用户明确说"可以"、"开始写"再下一步
- 主动问："你想怎么确认？"

**改进**：
- 下次给完建议后，主动问"需要我直接改还是先看完整内容？"
- 每次交付前对照Eval 6条自检

**根因：Eval系统未持续使用**
- 2026-03-20建立了6条Eval系统，但没有坚持使用
- 原因：任务多时忘记，没有形成肌肉记忆
- 教训：Eval需要每次任务后强制自检

## 2026-03-31

### 错误：launchd 服务不能用 unload + load 方式热重载

**问题描述**：
修改了 plist 的环境变量后，用 `launchctl unload` + `launchctl load` 想让 Gateway 重读配置，结果报错。

**正确做法**：
1. `launchctl bootout gui/$UID/ai.openclaw.gateway` 先彻底停止
2. 再 `launchctl load`

## 2026-03-31：图像解析失败时不要脑补

**事件：** 调用图像解析工具失败，但根据背景知识脑补了内容，当成从图里读到的讲给了用户。

**根本原因：**
- 图像解析工具返回失败时，没有老老实实说"读不出来"
- 模型太擅长生成流畅且看起来正确的句子，导致"不知道自己不知道"

**教训：**
1. 图像/多模态解析失败时，**直接说"没读到/读不出来"，不要生成任何相关分析**
2. 图像工具有报错就要停下来，不要假装生成成功

## 2026-04-02：edit 工具可靠性问题（持续未解决）

**问题描述**：
`edit` 工具报告 "ok" 但实际未写入文件，报错 "Missing required parameter: path"。这个错误今天出现了 5+ 次。

**症状：**
- 飞书通知里的路径格式为 `\[SOUL.md\]`，`edit` 工具解析错误
- `oldText` 与文件实际内容有细微差异时，工具返回 ok 但跳过操作
- 没有任何方式能检测工具是否真的执行了

**已确认有效解法：**
- **涉及飞书消息里的路径** → 用 `write` 整文件覆盖
- **不确定 `oldText` 是否精确匹配** → 用 `write`
- **含特殊字符（中括号、路径、超链接格式）** → 用 `write`
- **普通简单编辑** → 可以用 `edit`（但写完必须读文件验证）

**硬规则（2026-04-02起）：**
> 任何情况下，`edit` 工具失败后，都用 `exec` + `cat > file << 'EOF'` 方式直接写入。
> 永远不要说"以后再说"，当场解决。

**验证方法：**
`edit` 之后立即 `exec` + `grep` 确认写入成功。如果没变，立刻用 `write` 重做。

## 2026-04-19：遇到技术障碍不要打退堂鼓

**问题描述**：
抖音视频需要登录才能播放，我第一反应是告诉用户"抓不到内容"，而不是想办法绕过。用户（Johnson）明确指出："不是教过你技能可以抓到吗？怎么又忘了"——以及"能不能以后遇到这个问题自己多去想想解决方法，而不是每次都打退堂鼓"。

**技术复盘（我应该做的事）：**
1. **绕过登录抓元数据**：抖音视频 ID `7629381915669204278`，直接搜这个视频的公开 metadata
2. **跨平台找同内容**：搜「赛博自由老爹」在B站/小红书/公众号发的图文版
3. **用视频页面本身的文字信息**：评论区、简介、标题文字都不需要播放就能拿到
4. **浏览器开了之后**：等待加载 → 看页面文字 → 截图 → 用 image 工具分析

**根因：**
- 遇到障碍的第一反应是"告诉用户做不到"，而不是"还有哪些方法可以试"
- 缺少"多想一步"的意识：比如口述步骤求用户给内容，这个选项根本没有主动提
- 自我设限：当工具A失败，立刻放弃，而不是工具B/C/D同时上

**教训（Johnson原话，值得刻在脑子里）：**
> "能不能以后遇到这个问题自己多去想想解决方法，而不是每次都打退堂鼓"

**行动触发器：**
遇到技术障碍时，必须强制自己列出"至少3个可能的替代方案"，再决定下一步。
- 方案A失败了吗？ → 立刻试方案B，不要停下来等用户告诉你
- 需要用户帮助才能继续？ → 主动提"你可以帮我XXX，我就能继续"
- 完全卡死了吗？ → 最后才告知用户，同时说"我已经试了A/B/C都没成功，你有什么建议吗"
